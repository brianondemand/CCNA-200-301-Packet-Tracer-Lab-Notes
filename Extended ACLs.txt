Extended Access Control Lists (ACLs) are network security filters that evaluate and restrict traffic based on multiple granular criteria, including source IP address, destination IP address, specific protocols, and port numbers. 

They provide highly targeted network security that prevents unwanted traffic from crossing your infrastructure.


## Core Differences

Unlike Standard ACLs (which only filter based on source IP), Extended ACLs perform deep packet inspection using the following attributes:


Source IP: Who the traffic is coming from.

Destination IP: Where the traffic is trying to go.

Protocol: Layer 3 and 4 protocols like TCP, UDP, ICMP, or IP.

Port Numbers: Specific application ports (e.g., denying Telnet traffic on port 23, but allowing HTTP traffic on port 80)

## Identification and PlacementNumbered Lists: 


Ranging from 100–199 and the extended range 2000–2699.


Named Lists: Allows you to create rules by name and delete individual statements without having to recreate the entire list.


Best Practice Location: Always place Extended ACLs as close to the source of the traffic as possible. 


This helps conserve network bandwidth by dropping unauthorized packets early in the network path.



## Standard Configuration Example

1. Initiate named ACL: ip access-list extended [ACL-Name]

2. Define rule: [permit | deny] [protocol] [source] [source-wildcard] [destination] [destination-wildcard] [operator port]

3. Apply to interface: Navigate to the specific interface and use ip access-group [ACL-Name] [in | out]





Inbound and outbound ACLs (Access Control Lists) define the direction that network traffic is filtered as it crosses an interface. 

- Inbound ACLs filter packets before the device routes them, processing them immediately upon arrival. 


- Outbound ACLs filter packets after the device routes them, examining them just before they exit the interface.
